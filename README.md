# triad

`triad` is a deterministic verification kernel for one question:

> Is this claim true right now?

It keeps the product surface intentionally small. A `Claim` is the only canonical work unit. The reference surface is a pure core, a filesystem adapter, and a thin CLI.

## What It Includes

- `triad-core`
  Pure verification logic. No filesystem, config parsing, or process spawning.
- `triad-fs`
  Filesystem reference adapter for claims, snapshots, config, and evidence logs.
- `triad-cli`
  Reference CLI with exactly four commands: `init`, `lint`, `verify`, `report`.

Out of scope: `next`, `work`, `accept`, `agent`, runtime backends, and patch-draft workflow surfaces.

## Mental Model

- `Claim`
  One atomic contract you want to verify.
- `Evidence`
  Append-only verification records tied to a claim revision and artifact digests.
- `ClaimReport`
  The current result computed from the latest artifact snapshot plus stored evidence.

`ClaimStatus` has only five values:

- `confirmed`
- `contradicted`
- `blocked`
- `stale`
- `unsupported`

## What A Claim Is

A claim is one small statement that should be true in the repository right now.
Think of it as a verification target that sits above any single test case.

Examples:

- `REQ-auth-001`: valid login succeeds
- `REQ-api-003`: missing user returns `404`
- `REQ-billing-002`: failed payment does not finalize the order

A good claim is small enough that you can gather concrete evidence for it.

## Quick Start

```bash
cargo run -p triad-cli -- init
cargo run -p triad-cli -- lint --all
cargo run -p triad-cli -- verify --claim REQ-auth-001
cargo run -p triad-cli -- report --all --json
cargo run -p triad-cli -- --repo-root ../triad-repo report --all --json
```

## CLI Contract

- `triad init`
- `triad lint [--claim <CLAIM_ID> | --all] [--json]`
- `triad verify --claim <CLAIM_ID> [--json]`
- `triad report [--claim <CLAIM_ID> | --all] [--json]`

`triad` discovers `triad.toml` by walking ancestors from the current working directory.
`--repo-root <PATH>` bypasses discovery and uses `PATH` as the repo root. Relative paths resolve from the current working directory.

What each command is for:

- `init`
  Create the minimal triad scaffold for a repository.
- `lint`
  Check claim files and config contract without running verification commands.
- `verify`
  Run verification commands for one claim, append evidence, and return the current report for that claim.
- `report`
  Recompute reports from stored evidence plus the current artifact snapshot without running verification commands.

Operational rule:

- If you need the report that includes newly created evidence, run `verify` first and then `report`.
- Starting `verify` and `report` at the same time does not guarantee that `report` will observe the evidence being appended by `verify`.

`--json` writes direct JSON objects or arrays. There is no envelope wrapper.

## Config Shape

`triad.toml` is a strict v2 contract.

```toml
version = 2

[paths]
claim_dir = "spec/claims"
evidence_file = ".triad/evidence.ndjson"

[snapshot]
include = ["src/**", "tests/**", "crates/**", "triad.toml", "Cargo.toml", "Cargo.lock"]

[verify]
commands = ["cargo test --lib", "cargo test --tests"]
```

`verify.commands` accepts:

- legacy string entries
- structured object entries with `command`, optional `locator`, and optional `artifacts`

During `triad verify --claim <CLAIM_ID>`, `{claim_id}` and `{claim_path}` are expanded against the selected claim.

## Evidence Notes

- Only `Hard` evidence changes status.
- Freshness is checked only against the artifact subset recorded on the evidence row.
- The reference shell adapter records `0 => pass` and non-zero `=> fail`.
- `unknown` evidence is not auto-generated by the shell adapter, so `blocked` reports require seeded, manual, or non-shell evidence paths.

## Verification Gate

```bash
cargo fmt --all --check
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
python3 scripts/verify_artifacts.py
```

`scripts/verify_artifacts.py` is the artifact/surface gate. Runtime behavior stays covered by Rust tests.

Useful smoke checks:

```bash
cargo run -p triad-cli -- lint --all --json
cargo run -p triad-cli -- verify --claim REQ-auth-001 --json
cargo run -p triad-cli -- report --all --json
cargo run -p triad-cli -- --repo-root ../triad-repo report --all --json
```
